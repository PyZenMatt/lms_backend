
## 🎯 Cosa è SchoolPlatform?


SchoolPlatform è una **piattaforma di e-learning (LMS)** progettata per insegnanti che vogliono vendere corsi online e studenti che vogliono imparare.

Ho scelto un’architettura a tre livelli: frontend, backend e database.

Il frontend è una SPA in React con Vite(Vite non “gira” in produzione. Lo usi per GENERARE il frontend, non per servirlo. Vite è il tool che uso per sviluppare e buildare il frontend React. In sviluppo mi fornisce un dev server con hot reload, mentre in produzione genera asset statici ottimizzati che vengono serviti da Render.), serve l’interfaccia utente e invia richieste HTTP alle API Django.


Django espone endpoint REST, gestisce autenticazione, validazione e logica di business, e interagisce con PostgreSQL tramite ORM.

Le risposte tornano al frontend in formato JSON, che aggiorna lo stato e renderizza la UI.

È un sistema **end-to-end**, il che significa che gestisce:
- **Registrazione utenti** (chi sono gli studenti, chi sono i maestri?)
- **Catalogo corsi** (visualizza, ricerca, filtra i corsi disponibili)
- **Pagamenti** (studente compra il corso usando Stripe)
- **Accesso ai contenuti** (una volta pagato, lo studente accede alle lezioni)
- **Valutazione tra pari** (gli studenti possono valutare i lavori gli uni degli altri)
- **Wallet virtuale** (TeoCoin, una valuta interna per sconti e reward)
- **Dashboard** (ogni ruolo ha una visione personalizzata del sistema)

**In una frase:** È Udemy + Patreon + un sistema di NFT/token interno, tutto in uno.

## 👥 Chi Sono gli Attori?

### 1. **Studente**
- Si registra, verifica l'email
- Naviga il catalogo corsi
- **Compra un corso** usando Stripe (con carta di credito)
- Accede alle lezioni, guarda i video, svolge esercizi
- Può partecipare a **peer review** (valuta il lavoro di altri studenti)
- Guadagna **TeoCoin** come reward per attività (esercizi completati, review fatte)
- Vede il suo **wallet con il saldo TeoCoin**

### 2. **Maestro (Teacher)**
- Si registra, aspetta approvazione admin prima di insegnare
- Crea corsi, definisce prezzo in Euro
- **Vende i corsi** agli studenti (riceve una percentuale del pagamento)
- Gestisce le lezioni (pubblica contenuti, assegna esercizi)
- Vede gli **studenti iscritti** e il loro progresso
- Riceve notifiche quando qualcuno compra il corso
- Può **accettare o rifiutare uno sconto** che lo studente offre tramite TeoCoin
- Vede il suo **guadagno totale** (quanto ha incassato)

### 3. **Admin**
- Approva i maestri (verifica siano professionisti affidabili)
- Approva i corsi prima che vadano in vendita
- Vede statistiche globali (quanto fatturato totale, che corsi sono più venduti)
- Gestisce controversie tra studenti/maestri
- Configura il sistema (tasso di cambio TeoCoin, limiti, etc.)

## 💳 Flusso Pagamento — Il Cuore del Business

### Come Funziona l'Acquisto?

1. **Studente clicca "Compra"** → scatta il flusso
2. **Backend crea una sessione Stripe** (comunica a Stripe: "Questo è un ordine di €49.99")
3. **Studente viene reindirizzato a Stripe** (entra nel checkout sicuro di Stripe)
4. **Studente inserisce carta di credito** (Stripe gestisce la sicurezza, noi non teniamo dati sensibili!)
5. **Stripe elabora il pagamento** (verifiche fraud, 3D Secure se necessario)
6. **Stripe invia conferma al nostro backend** via webhook (come un SMS: "Pagamento riuscito!")
7. **Backend processa il pagamento:**
   - ✅ Iscrive lo studente al corso
   - ✅ Accredita il maestro
   - ✅ Registra la transazione nel database
   - ✅ Invia notifiche
8. **Webhook handler ritorna "OK"** a Stripe
9. **Studente ritorna al sito** → vede il corso nella lista

**Punto critico: Idempotenza**

Se il backend crashasse prima di rispondere a Stripe, Stripe avrebbe ritentato l'webhook. Senza protezione → iscrizione duplicata!

La soluzione? **Salvare l'ID della transazione Stripe nel database.** Alla prossima richiesta: "Ho già elaborato questo pagamento? Sì → salta tutto, ritorna OK."

## 🎁 Il Sistema TeoCoin — Incentivi Gamificati

### Che cosa sono i TeoCoin?

Sono una **valuta interna della piattaforma**, come punti fedeltà. Non sono cripto reali (per default), sono soprattutto **tracciati nel database**.

### Come si Guadagnano?

| Azione | Reward | Quando |
|--------|--------|--------|
| Completare lezione | 5 TEO | Al termine della lezione |
| Completare corso | 50 TEO | Quando tutti gli esercizi sono fatti |
| Valutare esercizio di un altro | 3 TEO | Quando la review è completata |
| Comprare corso (bonus) | 10 TEO | Subito dopo il pagamento |
| Accettare uno sconto dal maestro | 75 TEO | Quando il maestro accetta |

### Come si Usano?

**Sconto su Prossimo Acquisto**
- Studente ha 50 TEO
- Nuovo corso costa €50
- Studente offre al maestro: "Se mi dai uno sconto di €7.50 (75 TEO), ti do 75 TEO"
- Maestro può **accettare** (riceve 75 TEO come guadagno extra) o **declinare** (studente paga prezzo intero)

**Staking (Futuro)**
- Tieni i TeoCoin "bloccati" per X mesi
- Ricevi interesse (es. 5% annuo)
- È come un "certificato di deposito" virtuale

## 🏗️ Stack Tecnico — Gli Strumenti

### Frontend che Vede l'Utente
- **React** — Interfaccia interattiva (bottoni, form, liste)
- **TypeScript** — Codice più sicuro (catch errori prima)

L'utente vede tutto questo come **un sito web/app responsive** che funziona su telefono e desktop.

### Backend che Elabora
- **Django + Django REST Framework** — Il "cervello" del sistema
  - Riceve richieste dal frontend
  - Valida i dati
  - Accede al database
  - Ritorna risposte in JSON
  
- **PostgreSQL** — Il database
  - Salva utenti, corsi, pagamenti, transazioni
  - Garantisce integrità dati (niente dati sporchi!)

- **Redis** — Cache veloce
  - Salva sessioni, log, token
  - Serve come "message broker" per task asincroni

- **Celery** — Task asincroni
  - Invia email (non bloccare l'utente!)
  - Processa immagini
  - Aggiorna statistiche

### Servizi Esterni
- **Stripe** — Pagamenti
- **(Opzionale) Web3** — Blockchain per TeoCoin on-chain

## 🔐 Autenticazione — Come il Sistema Sa Chi Sei

### Il Problema
Il frontend è **stateless** (ogni refresh è "nuova pagina"). Come ricorda chi hai comprato?

### La Soluzione: JWT Token

1. **Accedi con email/password**
2. **Backend verifica** → credenziali corrette? Genera un **JWT token** (una stringa speciale)
3. **Token salvato** nel browser (localStorage)
4. **Ogni richiesta futura** include il token nell'header
5. **Backend legge il token** → "Ah, sei Mario!" → processa richiesta

**Cosa contiene il token?**
- Chi sei (ID utente, email)
- Che ruolo hai (student, teacher, admin)
- Quando scade (es. fra 1 ora)

**Se manca o scade?**
- Ricevi un "refresh token" (token di lunga vita per ottenere nuovo access token)
- Scambi il refresh token con un nuovo access token
- Continui a lavorare senza dover fare login di nuovo

## 📊 Come Sono Organizzati i Dati?

Il backend è diviso in **moduli** (o "app" in termine Django):

### `authentication`
- Login, registrazione, verifica email
- Gestisce JWT token

### `users`
- Profilo utente (avatar, biografia, bio)
- Assegna ruoli (student, teacher, admin)
- Approva maestri

### `courses`
- Modello Corso (titolo, descrizione, prezzo, maestro)
- Modello Lezione (video, testo, durata)
- Modello Esercizio (prompt, valutazione)
- Modello Iscrizione (chi è iscritto a quale corso)

### `payments`
- Integrazione Stripe
- Gestione webhook (quando un pagamento arriva)
- Idempotenza (niente duplicati!)

### `rewards`
- Contabilità TeoCoin (chi ha quanti TEO)
- Transazioni (quando i TeoCoin si muovono)
- Decisioni sconto (maestro accetta/declina)
- Staking

### `notifications`
- Sistema di notifiche event-driven
- "Hai completato la lezione!" 
- "Un nuovo studente si è iscritto!"

### `blockchain` (Opzionale)
- Web3 integration
- Mint TeoCoin on-chain
- Trasferire TEO su wallet reale

## 🎯 Principi Chiave del Design

### 1. **Separazione Responsabilità**
- Frontend visualizza, raccoglie input, invia al backend
- Backend valida, elabora, persiste, ritorna risultati
- Nessuno fa il lavoro dell'altro

### 2. **Idempotenza**
- Una azione può essere ritentata senza effetti collaterali
- Pagamento elaborato una sola volta, anche se webhook arriva 3 volte

### 3. **API First**
- Il backend espone API REST
- Qualsiasi frontend (web, mobile, CLI) può usare le stesse API

### 4. **Security by Default**
- JWT token firmati
- Webhook Stripe verificato
- Permission checks su ogni endpoint sensibile
- Password hashate, mai plaintext

### 5. **Scalabilità**
- Task asincroni (Celery) per non bloccare
- Cache Redis per letture frequenti
- Query database ottimizzate (select_related, prefetch_related)
